# 面试准备 — Cursor Rule Project

## 项目简介

基于 Cursor / 通义灵码的 **AI 编码规则共享仓库**，通过 `.mdc` 规则文件约束 AI 生成代码的行为，实现团队编码规范的自动化执行。覆盖 Java（Spring Boot）、Python、SQL 三个技术栈，支持 Cursor 一键导入和通义灵码兼容。

## 技术亮点

| # | 亮点 | 说明 |
|---|------|------|
| 1 | **规则即代码** | 将编码规范从文档转化为 AI 可执行的规则文件，规范落地不再依赖人工 Review |
| 2 | **双引擎兼容** | 同一套规范同时支持 Cursor 和通义灵码，通过桥接规则消除工具差异 |
| 3 | **分层触发机制** | 利用 `globs` + `alwaysApply` 实现规则按文件类型精准匹配，避免无关规则干扰 |
| 4 | **自愈式项目检查** | 缺失文档、依赖文件时自动生成，降低项目初始化成本 |
| 5 | **面试复盘自动化** | interview-review 规则可一键生成项目技术说明和面试题，辅助求职准备 |

## 技术难点

### 1. MDC 规则格式的设计与调试

- **问题**: Cursor 的 `.mdc` 格式缺乏官方详细文档，frontmatter 字段（`globs`、`alwaysApply`）的组合行为需要反复实验
- **解决**: 通过逐条测试确定：`globs` 为空且 `alwaysApply: true` = 全局生效；`globs` 有值且 `alwaysApply: false` = 按文件类型触发；两者都为空 = 仅手动或 AI 判断触发
- **方案**: 将测试结论固化为三层触发架构（全局层 / 语言层 / 按需层）

### 2. 跨 AI 工具的规则兼容

- **问题**: Cursor 和通义灵码的规则存储路径、格式完全不同，团队成员使用不同工具时规则无法统一
- **解决**: 设计桥接层 `lingma-rules.mdc`，Cursor 启动时主动扫描 `.lingma/rules/` 目录并合并执行
- **方案**: 冲突优先级明确（Cursor 规则 > 灵码规则），避免规则矛盾

### 3. 规则粒度的平衡

- **问题**: 规则太粗（如"写好代码"）AI 无法执行；太细（如"第3行必须空行"）则规则数量爆炸
- **解决**: 每条规则聚焦一个可判定的行为，配正确/错误对照代码示例
- **方案**: 规则按"原则 → 规则表 → 代码示例 → 例外情况"四段式结构编写

### 4. 导入失败排查

- **问题**: GitHub 导入报错 "No importable `.mdc` project rule files were found"
- **解决**: 排查发现 Cursor 只识别 `.cursor/rules/` 路径下的 `.mdc` 文件，原项目只有根目录 `.md` 文件
- **方案**: 将 `.md` 转为 `.mdc` 并放入正确路径，原始文档移入 `a-doc/` 保留

## 10 道面试题

### Q1：这个项目解决什么问题？

**考察点**: 项目理解、业务价值表达

**答案要点**:
- 解决团队编码规范难以落地的问题
- 传统方式靠文档 + Code Review 人工检查，效率低、覆盖不全
- 本项目将规范转化为 AI 可执行的规则，编码时实时约束，规范自动生效

### Q2：`.mdc` 文件的 frontmatter 中 `globs` 和 `alwaysApply` 如何配合工作？

**考察点**: 核心机制理解

**答案要点**:
- `alwaysApply: true` + `globs` 为空 → 所有文件操作都触发
- `alwaysApply: false` + `globs: "**/*.java"` → 仅编辑 Java 文件时触发
- 两者都为空 → 手动调用或 AI 根据上下文判断是否触发

### Q3：为什么要将规则按语言分层而不是写成一个大文件？

**考察点**: 架构设计、关注点分离

**答案要点**:
- 单一职责：每条规则职责明确，便于独立维护和开关
- 精准匹配：通过 `globs` 避免 Python 规则干扰 Java 编码
- 可组合性：不同项目可按需选择子集导入
- Token 效率：AI 上下文有限，只加载相关规则减少 token 消耗

### Q4：如何实现 Cursor 和通义灵码的规则兼容？

**考察点**: 跨平台兼容设计

**答案要点**:
- 两个工具规则路径不同：`.cursor/rules/` vs `.lingma/rules/`
- 设计桥接规则 `lingma-rules.mdc`，标记为 `alwaysApply: true`
- Cursor 启动时先读自身规则，再通过桥接规则扫描灵码目录
- 冲突时 Cursor 规则优先，保证确定性

### Q5：`java-null-check-rules` 为什么强制用 `Objects` 而不是直接 `== null`？

**考察点**: Java 基础、代码一致性

**答案要点**:
- `Objects.requireNonNull()` 提供 fail-fast 语义和清晰的错误信息
- `Objects::nonNull` 可作为方法引用用于 Stream 过滤，比 lambda 更简洁
- 统一风格降低 Review 成本，团队不再争论 `== null` vs `!= null` 的放置位置
- 本质是约定优于自由，不是技术优劣之分

### Q6：`spring-web-layered-rules` 中为什么禁止 Service 层返回 `Result<T>`？

**考察点**: 分层架构、职责边界

**答案要点**:
- `Result<T>`（code/msg/data）是接口协议，属于 Controller 层对外契约
- Service 层应返回纯业务对象，不关心 HTTP 响应格式
- 若 Service 返回 `Result`，则 Service 被 Controller 的协议绑定，无法被其他调用方（如 MQ 消费者、定时任务）复用
- 违反依赖倒置原则：高层模块的协议不应向下渗透

### Q7：`java-jdk-prefer-rules` 中列的替换是绝对的吗？什么时候可以用三方库？

**考察点**: 规则灵活性、工程判断

**答案要点**:
- 不是绝对的，规则明确列出三种例外：JDK 版本不支持、三方库有增强功能、项目已深度依赖
- 核心原则是"不引入不必要的依赖"，不是"消灭所有三方库"
- 例如 `StringUtils.substringBetween()` JDK 无等价实现，可以使用
- 新代码遵守规则，老代码不必为统一而大规模重构

### Q8：项目检查规则如何实现"缺失文件自动生成"？

**考察点**: 规则的可执行性

**答案要点**:
- `.mdc` 规则本质是对 AI 的指令，不是传统的 lint 程序
- `project-check-rules` 和 `python-project-rules` 中写明：缺失时"应自动生成"
- AI 在执行任务时读取规则，发现缺失会主动创建文件并填充模板内容
- 这是"声明式规则 + AI 执行"的模式，区别于脚本式自动化

### Q9：如果两条规则互相矛盾怎么办？你是怎么处理规则冲突的？

**考察点**: 系统设计、边界情况处理

**答案要点**:
- 同源冲突（Cursor 内部）：通过规则粒度控制避免，每条规则聚焦单一行为
- 跨源冲突（Cursor vs 灵码）：`lingma-rules.mdc` 明确声明 Cursor 优先
- 语言隔离：通过 `globs` 将不同语言规则隔开，物理上避免冲突
- 若仍有矛盾，AI 会提示用户决策，规则本身不做静默覆盖

### Q10：如果要将这套规则推广到一个 50 人的团队，你会怎么做？

**考察点**: 工程化思维、团队协作

**答案要点**:
- **仓库化**: 规则仓库独立维护，团队成员通过 GitHub 导入，更新即时同步
- **渐进式**: 先推核心规则（空值检查、分层架构），逐步增加，避免一次性规范过多引发抵触
- **可配置**: 支持按项目选择性导入规则子集，不强制全量
- **反馈闭环**: 收集误报/漏报案例，持续迭代规则内容
- **CI 集成**: 配合传统 lint 工具（Checkstyle、Pylint）做双重保障，AI 规则覆盖 lint 难以检查的架构约束
